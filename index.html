Sección 1: Arquitectura Fundamental de CPU (CP vs. zIIP)

El entorno de procesamiento en los mainframes IBM Z se basa en una distinción crítica entre dos tipos de procesadores. Comprender su interacción es fundamental para la economía y el rendimiento de z/OS Connect.

Procesadores de Propósito General (CPs)

Los CPs (o GCPs) son los motores de procesamiento primarios del mainframe. Son responsables de ejecutar el núcleo del sistema operativo z/OS, la lógica de aplicaciones tradicionales (como CICS, IMS, Db2) y la gestión de todas las operaciones de entrada/salida (I/O). El uso de CP está directamente vinculado a la clasificación de Millones de Unidades de Servicio (MSU) de la máquina, que es la base principal para los costos de licencia de software de Cargo de Licencia Mensual (MLC) tanto de IBM como de proveedores independientes (ISV). Por lo tanto, gestionar su utilización es un objetivo clave para la optimización de costos.

Procesadores de Información Integrada z (zIIPs)

Los zIIPs son motores de especialidad diseñados para procesar tipos de trabajo específicos y "nuevos", como Java y APIs RESTful, de forma asíncrona a los CPs. Su propuesta de valor es principalmente económica: el trabajo ejecutado en un zIIP generalmente no contribuye a los cargos de software basados en MSU. Esto permite a las organizaciones modernizar sus aplicaciones sin impactar significativamente sus costos de licenciamiento.

Una diferencia operativa clave es la velocidad. Los zIIPs siempre operan a la velocidad máxima del modelo de procesador, mientras que los CPs en modelos de subcapacidad son ralentizados deliberadamente. Esta disparidad es relevante cuando el trabajo destinado a un zIIP debe desbordarse hacia un CP.

El Modelo de Despacho CP-zIIP

El trabajo elegible para zIIP no se origina de forma autónoma en el zIIP. Primero debe ser reconocido y despachado por un CP. Un CP siempre está involucrado en la iniciación de la unidad de trabajo, que luego se transfiere a un zIIP disponible para su ejecución. Esta dependencia inicial significa que la disponibilidad y capacidad de respuesta de los CPs son un prerrequisito para la utilización eficiente de los zIIPs. El despachador de z/OS, junto con el Workload Manager (WLM), determina si la unidad de trabajo (técnicamente, un Bloque de Solicitud de Servicio o SRB) es elegible para ser descargada.

Sección 2: Anatomía de la Carga de Trabajo de z/OS Connect

z/OS Connect está construido sobre el perfil WebSphere Liberty y escrito predominantemente en Java. Esta base tecnológica es la clave de su perfil de rendimiento, ya que la ejecución de Java en z/OS es altamente elegible para la descarga a zIIP. IBM declara que más del 99% del código del producto z/OS Connect es elegible para ser descargado a procesadores zIIP.

Sin embargo, una transacción de API de extremo a extremo no es monolítica. Es una secuencia de tareas distintas, algunas vinculadas a CP y otras elegibles para zIIP.

Anatomía de un Flujo de API Provider (Entrante)

Este es el ciclo de vida de una solicitud de API que *entra* a z/OS para acceder a un sistema de registro (SoR) como CICS o IMS.

  1. Llegada y Despacho Inicial (CP-Bound): Cada solicitud entrante es manejada inicialmente por un CP, que gestiona la pila de red y prepara el trabajo para la JVM.
  2. Evaluación de Elegibilidad (CP-Bound): El despachador de z/OS y WLM determinan si el trabajo puede moverse a un zIIP.
  3. Procesamiento Principal (zIIP-Eligible): Si un zIIP está disponible, el trabajo se despacha allí. Esta es la parte más intensiva en CPU de z/OS Connect e incluye:
    • Transformación de Datos JSON: El análisis (parsing) de payloads JSON entrantes y la generación de JSON saliente es totalmente elegible para zIIP.
    • Seguridad y Criptografía: El cifrado/descifrado de red (IPsec) puede ser descargado a zIIPs.
  4. Llamada al Sistema de Registro (SoR) (CP-Bound): La llamada final al backend (CICS, IMS, Db2) se ejecuta predominantemente en CPs.
Punto Crítico: El costo total de CPU de una llamada API es la suma de (CPU de z/OS Connect) + (CPU del SoR). Mientras que el primer término es altamente elegible para zIIP, el segundo no lo es. No comprender esto puede llevar a un aprovisionamiento insuficiente de CPs.

Anatomía de un Flujo de API Requester (Saliente)

Este es el flujo cuando una aplicación z/OS (como CICS o un trabajo batch) llama a z/OS Connect para consumir una API RESTful externa.

  1. Llamada de Aplicación a z/OS Connect (CP-Bound): La comunicación inicial (ej. WOLA, IPIC) desde la aplicación z/OS a z/OS Connect está vinculada a CPs.
  2. Procesamiento Pre-Llamada (Mixto):
    • Seguridad (CP-Bound): La validación de la identidad (ej. SAF) es intensiva en CP, aunque puede mitigarse con perfiles RACLISTed.
    • Transformación de Datos (zIIP-Eligible): Convertir la estructura de datos de COBOL a un payload JSON (marshalling) es una actividad central de Java y altamente elegible para zIIP.
  3. E/S de Red Saliente (CP-Bound): El establecimiento del socket, el handshake TLS y el envío de la solicitud HTTP son funciones del sistema operativo vinculadas a CPs.
  4. Espera de Respuesta: El hilo espera, consumiendo mínima CPU.
  5. Procesamiento Post-Llamada (Mixto):
    • Recepción de Red (CP-Bound): Recibir la respuesta HTTP está vinculado a CPs.
    • Transformación de Datos (zIIP-Eligible): Analizar el JSON de respuesta y transformarlo de nuevo a COBOL (unmarshalling) es highly elegible para zIIP.
  6. Retorno a la Aplicación (CP-Bound): Devolver los datos a la aplicación z/OS llamante está vinculado a CPs.
Punto Crítico: El rendimiento de una llamada de API Requester es una secuencia de tareas. Un cuello de botella en las fases vinculadas a CP (como la E/S de red o la seguridad) puede dominar el tiempo de respuesta total, incluso si la utilización de zIIP es baja.

Sección 3: El Impacto de la Escasez de Procesadores

La contención de procesadores ocurre cuando la demanda de ciclos de CP o zIIP excede la oferta disponible. Esto no es un problema aislado; desencadena una cascada de efectos que degradan el rendimiento.

Definición de Contención y Retardo de CP

WLM gestiona la contención priorizando el trabajo. Cuando una unidad de trabajo está lista para ejecutarse pero no hay un CP disponible, incurre en "Retardo de CPU" (CPU Delay). Esta es una métrica clave capturada por RMF y es el porcentaje de tiempo que un trabajo estuvo listo pero en cola, esperando un procesador. El Retardo de CP se traduce directamente en tiempos de respuesta aumentados e impredecibles, rendimiento reducido e inestabilidad del sistema.

Escenario 1: Saturación de zIIP y el Efecto "Spillover" (Desbordamiento)

Los zIIPs son un recurso finito. Cuando todos los zIIPs disponibles están ocupados (debido a altos volúmenes, JSON grandes o cargas de trabajo concurrentes), el sistema redirige el trabajo elegible para zIIP para que se ejecute en los CPs de propósito general. Este "desbordamiento" (spillover) tiene graves consecuencias:

  • Aumento de Costos de Software: El trabajo que estaba destinado a ser "gratuito" (desde la perspectiva de licenciamiento) ahora se ejecuta en CPs facturables, consumiendo MSUs y aumentando los costos de MLC.
  • Aumento de la Contención de CP: Este trabajo de desbordamiento ahora compite directamente con el trabajo crítico que *debe* ejecutarse en CPs (como CICS, Db2 y la E/S de red).
  • Degradación del Rendimiento: El trabajo desbordado perjudica activamente el rendimiento de toda la LPAR. Ralentiza el SoR de backend, lo que a su vez retiene la solicitud de z/OS Connect por más tiempo, exacerbando el problema en un ciclo de retroalimentación de degradación.

Escenario 2: Escasez de CP y el Efecto "Bottleneck" (Cuello de Botella)

Un entorno con restricciones de CP es aquel en el que la demanda total de ciclos de CP (trabajo de SoR, E/S, gestión del sistema operativo y trabajo desbordado de zIIP) excede la capacidad. Esto paraliza a z/OS Connect de varias maneras:

  • Retraso en el Manejo Inicial: El primer paso de recibir una solicitud de la red se retrasa, ya que requiere un CP.
  • Llamadas al SoR/Red Congeladas: Todas las llamadas salientes (al SoR en flujos de Provider, a la red en flujos de Requester) deben despacharse en un CP. La escasez significa esperas más largas para estas partes críticas.
  • Gestión de JVM Degradada: La Recolección de Basura (Garbage Collection - GC) puede tener componentes que requieren ciclos de CP. Los retrasos aquí provocan pausas en la aplicación.

Sintomatología y el Ciclo de Retroalimentación

La escasez de CP crea un ciclo de retroalimentación perjudicial, especialmente para los flujos de API Requester:

  1. El Retardo de CP ralentiza las tareas vinculadas a CP (ej. E/S de red).
  2. Esto aumenta la duración total de la solicitud, lo que significa que las hebras (threads) de Liberty se retienen por más tiempo.
  3. Una mayor ocupación de hebras aumenta la concurrencia y la probabilidad de saturación de zIIP (más transformaciones JSON activas).
  4. La saturación de zIIP causa desbordamiento (spillover) a los CPs.
  5. Este desbordamiento aumenta aún más la contención de CP, regresando al paso 1 con un retardo aún peor.

Este ciclo puede llevar a una rápida degradación del servicio, agotamiento de la agrupación de hebras y fallos en cascada. Un valor de desbordamiento alto no es solo un problema de costos; es un indicador crítico en tiempo real de la degradación del rendimiento sistémico.

Sección 4: Marco de Medición y Análisis

Para diagnosticar estos problemas, es indispensable un enfoque de monitoreo de múltiples capas, correlacionando datos del sistema (RMF) con datos de la aplicación (SMF 123).

Capa 1: Análisis a Nivel de LPAR (RMF/SMF Tipo 70)

Este informe proporciona una visión de alto nivel de la salud general de los procesadores de la LPAR.

  • Informe Clave: Informe de Actividad de CPU de RMF (SMF Tipo 70, Subtipo 1).
  • Métricas a Analizar:
    • LPAR Busy vs. MVS Busy: Si LPAR Busy es significativamente mayor, indica contención a nivel de CEC (demanda latente).
    • Ocupación de CP y zIIP del LPAR: Buscar intervalos donde la ocupación de CP sea consistentemente alta (>95%) y la ocupación de zIIP también sea alta. Esta combinación predice el desbordamiento.
    • Vista de HiperDispatch: Verificar si hay procesadores "Aparcados" (Parked). Si no hay procesadores aparcados durante picos, indica alta utilización.

Capa 2: Análisis Granular de Carga de Trabajo (RMF/SMF Tipo 72)

Este es el informe más crítico para entender la efectividad de la descarga a zIIP y medir la contención a nivel de la clase de servicio de z/OS Connect.

  • Informe Clave: Informe de Actividad de la Carga de Trabajo de RMF (WLMGL) (SMF Tipo 72, Subtipo 3).
  • Métricas Clave a Analizar:
    Métrica Campo SMF 72-3 Descripción Qué Buscar (Interpretación)
    APPL% -> CP - Porcentaje de tiempo de CPU que se ejecutó en un CP estándar. Indica trabajo no elegible o desbordado.
    APPL% -> IIP - Porcentaje de tiempo de CPU que se ejecutó con éxito en un zIIP. Mide la descarga exitosa. Un valor alto es deseable.
    APPL% -> IIPCP R723SUCU Porcentaje de tiempo de CPU que era elegible para zIIP pero se vio forzado a ejecutarse en un CP estándar debido a que los zIIPs estaban ocupados. La métrica más importante para la saturación de zIIP. Un valor creciente es un indicador principal de un cuello de botella de zIIP.
    CPU DLY % R723CADL Porcentaje de tiempo que la carga de trabajo estuvo lista para ejecutarse pero estaba esperando un CP. La medida directa del retardo de CP. Causa raíz del aumento del tiempo de respuesta.
    VELOCITY - Indicador de si la carga de trabajo está cumpliendo sus objetivos de WLM. Una velocidad por debajo del objetivo indica que la carga de trabajo está siendo retrasada.

Capa 3: Diagnóstico en Tiempo Real (RMF III / OMEGAMON)

Estas herramientas son cruciales para diagnosticar problemas activos e identificar los trabajos específicos que causan contención.

  • RMF Monitor III:
    • Informe de Retrasos de Trabajos (Job Delays): Muestra qué espacios de direcciones están siendo retrasados. Un valor alto en la columna PRC para z/OS Connect indica que está esperando un procesador.
    • Informe de Retrasos del Procesador (Processor Delays): Desglosa el retraso PRC. Un DELAY% alto para zIIPs significa saturación de zIIP. Un DELAY% alto para CPs significa contención de CP.
  • OMEGAMON for JVM (o similar):
    • Uso de CPU de JVM: Muestra el desglose en tiempo real entre tiempo de CP, tiempo de zIIP y, crucialmente, el trabajo elegible para zIIP que se ejecuta en CPs (el equivalente en tiempo real de IIPCP).
    • Detalles de Hebras (Threads): Permite identificar hebras bloqueadas o de larga duración.
    • Alertas (Situaciones): Se deben configurar alertas proactivas.
      • Advertencia: % zIIP en CP > 10% sostenido.
      • Crítico: % zIIP en CP > 50% (indica contención severa).
      • Crítico: Tiempo de Respuesta Promedio > 2x la línea base.
      • Advertencia: Recuento de Hebras Activas > 80% del máximo (riesgo de agotamiento).

Capa 4: Análisis de Solicitudes de Extremo a Extremo (SMF Tipo 123)

Este registro es generado por el propio z/OS Connect y es la fuente definitiva para el tiempo de respuesta de las solicitudes individuales. Es crucial habilitar los registros V2 (apiRequesterSmfVersion="2") para flujos de Requester, ya que contienen marcas de tiempo más detalladas.

  • Metodología de Correlación:
    1. Identificar períodos de alto Retardo de CP (CPU DLY %) o alto desbordamiento (IIPCP %) usando RMF (SMF 72).
    2. Extraer los registros SMF 123 para el mismo intervalo de tiempo.
    3. Calcular los componentes del tiempo de respuesta usando las marcas de tiempo de SMF 123.
  • Cálculo de Tiempos (Ejemplo API Provider):
    Marca de Tiempo (V2) Descripción Fórmula
    TIME_ZC_ENTRY Llegada de la solicitud a z/OS Connect. Tiempo en SoR = TIME_SOR_RECV - TIME_SOR_SENT

    Tiempo total en z/OS Connect = TIME_ZC_EXIT - TIME_ZC_ENTRY

    Tiempo de procesamiento en z/OS Connect = (Tiempo total) - (Tiempo en SoR)
    TIME_SOR_SENT Solicitud enviada al SoR.
    TIME_SOR_RECV Respuesta recibida del SoR.
    TIME_ZC_EXIT Respuesta enviada al cliente.
  • Conclusión de la Correlación: Si el "Tiempo de procesamiento en z/OS Connect" (o "Tiempo en z/OS Connect" para flujos de Requester) está elevado durante períodos de alto CPU DLY % o IIPCP %, se demuestra que la contención de CPU es la causa raíz de la degradación del rendimiento.

Sección 5: Estrategias Proactivas de Ajuste y Optimización

El diagnóstico debe ir seguido de acciones correctivas. Estas estrategias ayudan a prevenir la contención y asegurar un rendimiento óptimo.

1. Configuración Estratégica de WLM

Una configuración adecuada de WLM es la primera línea de defensa. No se debe usar un objetivo discrecional para z/OS Connect.

  • Clasificación: Es fundamental habilitar zosWlm-1.0 en server.xml y usar reglas de httpClassification para asignar la STC del servidor y los enclaves de solicitud (tipo CB) a clases de servicio separadas y de múltiples períodos.
  • Objetivos de Múltiples Períodos: Para las solicitudes (enclaves CB), el primer período debe tener un objetivo de velocidad alto (ej. Velocity 80%) con una duración corta. Esto prioriza las solicitudes rápidas. Las solicitudes de larga duración caen a períodos posteriores con menor prioridad, protegiendo al sistema.
  • Niveles de Importancia (Flujo Provider): Para un proveedor de API, el SoR de backend (ej. CICS) debe tener una importancia más alta que el servidor z/OS Connect. Esto actúa como un estrangulamiento natural, evitando que z/OS Connect inunde un backend con dificultades y protegiendo al componente más vulnerable.
  • Niveles de Importancia (Flujo Requester): Para un solicitante de API, el servidor z/OS Connect debe tener una importancia más alta que la aplicación que lo llama (ej. un trabajo batch).
  • Parámetro IIPHONORPRIORITY: Este parámetro de LPAR (predeterminado YES) permite que el trabajo de zIIP desbordado compita por CPs según su prioridad de WLM. En entornos con escasez severa de CP, cambiarlo a NO (forzando al trabajo a esperar un zIIP) podría ser una opción de ajuste avanzada para preservar los CPs, aunque puede aumentar la latencia.

2. Ajuste del Subsistema z/OS y TCP/IP (Enfoque Requester)

Dado que los flujos de solicitante son intensivos en red, optimizar TCP/IP puede reducir significativamente el consumo de CP y la latencia.

  • Habilitar AsyncIO: Habilitar el servicio nativo de z/OS AsyncIO permite a Liberty un manejo de sockets más eficiente, mejorando el rendimiento y la escalabilidad de las llamadas salientes.
  • Ajuste del Perfil de TCP/IP:
    • TCPCONFIG DELAYACKS: Habilitar acuses de recibo retardados puede reducir la cantidad de paquetes y el consumo de CPU.
    • Checksum Offload: Descargar los cálculos de suma de comprobación a las tarjetas de red OSA-Express reduce la carga en los CPs.
    • Tamaños de Búfer: Asegurar tamaños adecuados (ej. >= 64KB) para DefaultSendBufferSize y DefaultReceiveBufferSize optimiza el rendimiento de la red.

3. Optimización a Nivel de Aplicación (Reducción de Payloads)

Reducir la demanda de CPU en la fuente es la forma más eficiente de optimización. El costo de CPU es directamente proporcional al tamaño del payload JSON que debe ser transformado.

  • Excluir Campos No Utilizados: Trabajar con los desarrolladores para mapear solo los campos absolutamente necesarios. Excluir campos no utilizados del mapeo del copybook reduce drásticamente el tamaño del payload y el consumo de CPU.
  • Usar OCCURS DEPENDING ON (ODO): Para arrays de COBOL, usar ODO en lugar de arrays de tamaño fijo permite payloads de tamaño variable, reduciendo la sobrecarga de transformación.
  • Renombrar Campos: Usar nombres de campo más cortos en el esquema JSON reduce los datos a analizar, proporcionando una ganancia marginal pero significativa para APIs de alto volumen.

4. Planificación Proactiva de la Capacidad

La planificación de capacidad es fundamental y no debe basarse en promedios, sino en volúmenes máximos, ya que las cargas de trabajo de zIIP suelen ser "puntiagudas". El objetivo debe ser mantener el valor de IIPCP cerca de cero durante los picos de negocio.

Las métricas % CPU DLY y % IIPCP deben guiar la toma de decisiones:

  • % CPU DLY alto, % IIPCP bajo: Cuello de botella clásico de CP. La solución es añadir CPs o ajustar otras cargas de trabajo pesadas en CP.
  • % CPU DLY bajo, % IIPCP alto: Cuello de botella clásico de zIIP. La solución es añadir zIIPs.
  • % CPU DLY alto, % IIPCP alto: El sistema está completamente saturado. Este es un déficit crítico de capacidad que requiere la adición de ambos, CPs y zIIPs.
  • % CPU DLY bajo, % IIPCP bajo: El problema de rendimiento no está relacionado con la contención de procesadores en z/OS. La investigación debe centrarse en la latencia de la red, el rendimiento del endpoint remoto o la lógica de la aplicación.

Un valor de % IIPCP del 100% significa que un CP completo se está utilizando para trabajo de zIIP. Esto se puede traducir directamente en un impacto de costo de MLC, proporcionando una poderosa justificación financiera para adquirir motores zIIP adicionales.

Sección 6: Conclusión y Resumen de Hallazgos

El rendimiento de z/OS Connect está intrínsecamente ligado a la gestión eficaz de los procesadores CP y zIIP. La arquitectura, impulsada por consideraciones económicas, ofrece un enorme potencial para procesar cargas de trabajo de modernización de manera rentable. Sin embargo, este modelo introduce un riesgo crítico: la saturación de los procesadores zIIP y la contención de los CPs.

El análisis ha demostrado que la carga de trabajo de z/OS Connect es una amalgama de tareas vinculadas a CP (E/S de red, seguridad) y tareas elegibles para zIIP (transformación JSON). La escasez en cualquiera de los procesadores crea un efecto dominó:

  • La escasez de CP retrasa las tareas de red y seguridad, aumentando el tiempo total de la solicitud.
  • Esto aumenta la concurrencia, lo que lleva a la saturación de zIIP.
  • La saturación de zIIP causa un desbordamiento (spillover) de trabajo Java hacia los CPs.
  • Este desbordamiento exacerba la contención de CP, degradando el rendimiento de todo el sistema y aumentando los costos de licenciamiento.

La clave del éxito reside en una gestión proactiva y basada en datos. Es imperativo implementar un marco de monitoreo que correlacione los datos de RMF (SMF 70 y 72) con los registros de aplicación (SMF 123). La métrica IIPCP (o % zIIP en CP) emerge como el indicador principal más crucial de la inminente saturación de zIIP y la degradación del rendimiento.
